产品方向评审

Isometry 是否应该成为 Agent 交易的责任层？

我们的候选产品不是让 Agent 更容易付款，而是让普通人和机构敢把付款委托给 Agent：交易前把意图变成可裁决条件，交易中拥有否决权，交易后按证据先赔后追。

候选定位：跨支付轨道的委托承保层——编译意图、执行边界、签发保障、路由责任。

为什么不是“再做一个 Permit”

Mandate、受限支付 Token、消费控制已在快速标准化。真正未被验证的产品赌注，是谁愿意把这些零件组合成可承保、可理赔、可追偿的责任产品。

Token / Mandate / Policy ≠ Responsibility

方向成立的必要条件

1
必须拥有否决权不能只给风险分；被承保交易必须在结算前经过 Isometry。
2
只承保可裁决条款“金额 ≤ 400 美元”可以；“体验好”默认不可以。
3
赔付与追偿均可运行谁先赔用户与谁最终承担损失，需要分开设计。
核心结论：我们不是在信用卡与稳定币之间选管道，而是在所有管道之上建立 Agent 交易的委托承保与责任层。方向值得验证，但 Permit 如何产生、承保到哪一层、谁提供风险资本、法律上以什么身份承诺，均未拍板。

      What is actually new

      创新不应建立在“别人没有 Permit”上

      目前市场事实要求我们把差异化向责任、运营与资产负债表推进，而不是停在 Token 或协议层。

      已经在发生：不能当作独占创新

      行业已经在标准化 Agent 的 Mandate、范围受限的支付凭证、消费控制、认证和可追踪性。

      • 用户意图与预授权条件可以被记录为可验证 Mandate。
      • 支付凭证可以绑定商户、时间、金额并隐藏底层卡信息。
      • 网络层已加入 Agent 身份、认证、消费控制与争议信号。

      候选创新：把零件做成责任产品

      Isometry 的产品赌注应是：只为自己能验证和执行的条款签发保障，并承担理赔、垫付和追偿运营。

      • Coverage Certificate：明确“什么错由谁赔”。
      • Hard Gate：承保前拥有交易否决权。
      • Evidence Packet：让裁决和追偿可自动化。
      • Risk Capital：准备金、限额、定价与再担保。
      外部事实参考（官方来源）：Google AP2 / Mandates；Stripe Shared Payment Tokens；Visa Intelligent Commerce。这些来源说明“授权与控制”正在成为基础设施；本报告据此把产品赌注推向承保与 Remedy，而非功能罗列。
      SCAN
      另附：Canton、Tempo 等方案的责任层缺口扫描结论不是“他们什么都没有”：Canton 可承载权利义务与证据；Tempo、Anyway、Stripe 和卡网络已有不同程度的控制、Trace 与争议能力。真正要判断的是谁公开签字承保、谁先赔、谁承担追偿失败。

      Permit & underwriting

      谁定义 Permit，为什么 Isometry 赔

      正确答案不是“Permit 全由 Isometry 定”，而是分清经济意图、执行语义与承保承诺。

      用户 / 企业决定

      Intent Mandate

      用户定义目的、预算和偏好；企业追加供应商、审批、税务和预算政策。Isometry 不替用户决定买什么。

      待拍板：谁编译

      Execution Permit

      把自然语言和组织政策转换为机器谓词。候选模式包括用户直接配置、Isometry 编译后确认、或平台模板。

      Isometry 决定是否签字

      Coverage Certificate

      Isometry 只为有控制权、有证据、有清晰 Remedy 的条款承保。它赔的理由，是自己主动签了这份保障。

      一个可承保条款必须长这样

      { "clause": "final_total <= 400 USD", "author": "user", "source_of_truth": "final_capture", "validator": "isometry", "evaluation_time": "payment_authorization", "tolerance": "0 USD", "guarantor": "isometry", "remedy": "reimburse_excess", "liability_cap": "400 USD" }
      Predicate什么算通过或违反；必须是确定的机器表达。
      Source of truth以商户签名报价、最终 Capture、Issuer 认证还是第三方数据为准。
      Evaluation time在付款授权、Capture、履约时还是事后某一时点判断。
      Tolerance价格漂移、时间偏差、分类置信度允许多少。
      Guarantor & remedy谁签字、违反时赔超额、赔全单、垫退款还是只协助争议。
      硬约束：可直接承保

      金额、商户、SKU、时间、数量、审批、Permit 有效期。

      可验证事实：条件承保

      报价时计划到达时间、退款条款、库存；必须指定事实源和时点。

      优化目标：需定义基准

      “最低价”要先定义比较集合、窗口和总价口径，才能裁决。

      主观偏好：默认不承保

      性价比高、舒服、靠谱、有氛围、用户事后不满意。

      因此应删除“Agent 违反用户预期”这一责任项。准确分类应是：受保障条款被违反、优化目标未达、商户履约失败或用户偏好不满意。
      Card fraud integration

      现有信用卡欺诈风控：必须接，但不能代替 Isometry

      系统化答案不是“接一个欺诈分数”，而是把资金风险、委托风险、订单风险和争议追偿拆开。

      信用卡风控擅长识别盗卡、账户接管、设备与行为异常；它不天然知道“这个合法持卡人的 Agent 是否被允许买这一张票”。Isometry 不应重做 Issuer，而应把 Agent 的任务、Permit、Quote 和证据补进现有支付网络。

      原则：双方都有否决权

      谁先赔用户

      是产品体验问题。Isometry 可用准备金即时垫付，让用户不必等待商户、PSP 或卡组织完成漫长争议。

      谁最终承担损失

      是证据与合同问题。Isometry 根据失败来源向商户、PSP、Issuer、数据方或自身风险池归集损失。

      Stablecoin verdict

      要不要用稳定币：底层可以必须用，用户不应必须用

      这是目前最明确的架构结论之一。

      多轨执行，统一责任标准

      消费者继续用银行卡，商户继续收法币；Isometry 可以在内部使用稳定币提升赔付、结算和资金控制效率。稳定币是责任产品的资金工具，不是用户入口。

      稳定币服务于 Guarantee；不能让 Guarantee 反过来服务于发币。
      赔付准备金

      为已签发 Coverage Certificate 配置可实时计量的风险敞口和流动性。

      即时垫付退款

      先让用户拿回钱，再走商户退款、网络争议或追偿。

      多方内部结算

      商户、平台、Isometry、担保方之间更快清分。

      可编程资金控制

      Escrow、hold、release、限额与风险池隔离。

      用户不需要做的事

      不必买稳定币、创建钱包、保管私钥、理解 gas、选择链。若这些成为前提，产品又回到了“开发者 ∩ crypto”的窄圈。

      Product truth test

      方向成立必须证明什么；什么情况应停止

      下一步不是做市场地图或推广，而是验证责任产品是否真实可运行。

      必须为真

      1. 常见委托能被可靠地编译成机器可裁决条款，并由用户理解和确认。
      2. Isometry 能在被承保交易结算前拥有硬否决权。
      3. 证据足够判断失败来源，并能进入商户退款、支付争议或追偿流程。
      4. 责任转移本身有价值，而不是客户只需要一个消费限额控件。
      5. 赔付敞口可通过限额、商户分层、准备金与再担保被管理。

      停止或转向的信号

      • 拿不到交易闸门，只能事后评分或观察。
      • 主要投诉集中在“感觉不满意”等不可裁决问题。
      • 商户与支付网络不给足够证据，导致无法追偿。
      • Coverage 只能靠大量人工、无法形成稳定规则和定价。
      • 网络/PSP 把同样保障无成本捆绑，Isometry 没有独立控制面或数据优势。

      Decision requested

      建议：批准“继续验证方向”，而不是批准一个已经定型的产品

      目前最有价值的结论，是把问题从“Agent 怎么付钱”推进到“谁有权让它付、谁敢为边界签字、出错后如何赔和追”。但 Permit、承保深度、风险资本和法律结构仍决定这个方向是否成立。

      建议认可的红线：不做 crypto-only 钱包；不把主观预期写进赔付；没有交易否决权和证据就不承保；不重造信用卡欺诈网络。

      建议的三种决策选项

      A. 方向成立，进入产品验证同意责任层定位和四条红线；开放问题保持未决，以一条完整责任链继续设计。
      B. 方向有价值，但需要收窄只验证 Enforcement Guarantee，不碰商户履约、Outcome Cover 或大额风险资本。
      C. 不成立如果 Isometry 无法取得闸门、承保与追偿能力，则不应把它包装成 Agentic Payment 产品。
